## 二、产品概述

### 2.1 产品定位

TSCS（Traffic Safety Coordination System，交通安全统筹系统）企业版是一个面向集团化运输企业的**第三方安全管理与互助统筹平台**。

**核心定位**：

- 服务客户：母公司旗下的多家运输子公司（出租车、租赁、客运旅游等）
- 覆盖规模：约30,000辆运营车辆、约50,000名驾驶员
- 平台本质：不是保险公司，也不是保险经纪平台，而是独立的**第三方安全统筹管理服务商**

**平台本质说明**：

本平台为互助共济性质，受《民法典》合同编约束，非商业保险业务。平台不承担保险赔付责任，而是通过管理统筹资金池、运营安全管理体系，为参保企业提供风险互助保障。统筹资金来源于各子公司缴纳的统筹费，互助给付款从统筹资金池中支付。

**目标价值链**：

```
运输企业（子公司）
  ↓ 缴纳统筹费
TSCS平台（第三方服务商）
  ↓ 安全管理 + 风险互助
  ↓ 出险时提供互助给付
驾驶员（被保障对象）
  ↓ 安全驾驶 + 参与安全培训
降低整体风险水平 → 费率下降 → 多方共赢
```

### 2.3 解决的核心问题

**问题一：子公司保险/统筹费用高、管理分散、数据不透明**

当前各子公司各自购买商业保险或参加不同统筹，费用标准不统一、议价能力弱、管理方式各异。集团无法实时掌握各子公司的保险/统筹费用支出、到期情况、保障范围等关键信息，导致重复采购、保障缺口、费用浪费等问题频发。

**问题二：集团无法统一掌握各子公司风险状况**

各子公司分散管理，出险数据、安全数据、驾驶员信息各自为政。企业层面缺乏统一的风险视图，无法进行跨分公司的风险对比、趋势分析、预警管理，导致高风险子公司和高风险驾驶员无法被及时发现和干预。

**问题三：出险处理效率低，缺乏数字化全流程管理**

当前出险处理依赖人工跟进，事故申报、核查、损失核定、互助理算、支付各环节信息割裂，处理周期长、效率低、易出错。分司管理员无法实时跟踪案件进度，驾驶员无法了解补偿进展，导致满意度低、纠纷多。

**问题四：驾驶安全管理缺乏数据支撑**

缺乏驾驶行为数据采集和分析体系，无法客观评价驾驶员安全水平。安全管理依赖事后处理而非事前预防，无法识别高风险驾驶员并采取针对性措施，也无法将安全表现与费率激励挂钩。

**问题五：资金管理不透明**

统筹资金缺乏透明化管理，各子公司缴纳的统筹费使用情况不清晰，互助给付款的计算和支付缺乏标准化流程，资金池的风险状况无法实时监控，影响企业和驾驶员对平台的信任。

### 2.4 产品目标

1. **实现企业级统筹管理数字化**：为30,000辆运营车辆提供统一的参保、事故申报、核查、补偿全流程数字化管理，覆盖企业入驻、批量参保、日常运营、续保退统全生命周期。

2. **构建多租户数据隔离体系**：支持集团旗下多家子公司独立运营、独立结算、数据隔离，同时同时为企业高管提供本企业全局数据视图（仅限本企业内数据，不能跨企业查看），实现"分散经营、统一管理"。

3. **建立驾驶安全评分体系**：基于车载终端离线回传数据和出险记录，建立科学的驾驶评分体系（0~100分），覆盖约50,000名驾驶员，为风险管理提供数据支撑。

4. **实现企业费率精准定价**：基于企业风控系数（历史赔付率、驾驶评分均值、续约年限、挂靠比例）和规模折扣，实现差异化费率定价，激励企业提升安全管理水平。

5. **提供全流程案件处理能力**：事故申报到结案全链路数字化，支持多渠道事故申报（驾驶员小程序、企业端、平台端）、分级审批、企业确认、自动互助理算，案件平均处理周期控制在15个工作日内。

## 三、目标用户与角色

### 3.1 角色总览表

| 角色 | 规模 | 主要终端 | 核心任务 | 所属端 |
|------|------|---------|---------|--------|
| 分司管理员 | 30~60人 | 企业端Web | 本分司范围的车辆管理、驾驶员管理、财务管理、案件查看 | 企业端 |
| 安全员 | 20~40人 | 企业端Web | 车辆/驾驶员查看、安全看板、事故申报审核 | 企业端 |
| 高管 | 5~10人 | 企业端Web | 本企业全局看板、跨分司对比、关键审批（退统/大额确认） | 企业端 |
| 平台调度员 | 5~10人 | 平台端Web | 事故申报受理、案件分类、任务分配、进度跟踪 | 平台端 |
| 现场核查员 | 20~40人 | 现场端App | 接收任务、到达签到、现场拍照、笔录记录、损失评估 | 现场端 |
| 互助给付员 | 5~10人 | 平台端Web | 材料审核、互助理算、分级审批、给付款支付 | 平台端 |
| 平台财务人员 | 3~5人 | 平台端Web | 收款确认、独立结算对账、退费处理、报表生成 | 平台端 |
| 平台运营人员 | 3~5人 | 平台端Web | 企业入驻审核、费率配置、保障方案管理、系统配置 | 平台端 |
| 驾驶员（被保障驾驶员） | 约50,000人 | 驾驶员端小程序 | 事故申报、案件进度查询、驾驶评分查看、消息通知 | 驾驶员端 |

**角色详细说明**：

**分司管理员**：各子公司指定的日常运营管理人员，负责本分司范围内的车辆管理（自有+挂靠）、驾驶员分配、参保操作、案件查看、财务对账等日常工作。每个子公司可设置多名分司管理员。分司管理员不参与安全审核和全局看板查看。

**安全员**：各子公司指定的安全管理岗位人员，负责本分司的车辆/驾驶员信息查看、安全看板查看、事故申报信息审核（完整性、真实性、合规性）。安全员是事故申报从企业端到平台端的必经审核环节。安全员不参与车辆管理、财务操作等日常运营。

**高管**：企业高管角色，可查看**本企业**内所有分司的全局统计数据、风险预警、跨分司对比分析报表。高管可进行关键审批操作（退统审批、大额互助给付确认等）。高管只能查看本企业内运营车辆，不能跨企业查看。高管不参与日常运营操作。

**平台调度员**：TSCS平台运营团队的核心岗位，负责接收各类渠道的事故申报信息（驾驶员小程序、企业端审核通过后、电话转派），进行案件分类、核查任务分配、案件进度跟踪。调度员是连接申报人和核查员的枢纽。

**现场核查员**：外勤人员，负责事故现场勘查、拍照取证、当事人笔录、初步定责、损失评估。核查员需使用现场端App完成移动办公，现场采集的数据直接回传平台。

**互助给付员**：TSCS平台核心后台岗位，负责审核案件材料完整性、计算互助给付款（互助理算）、发起分级审批、跟踪支付进度。互助给付员需具备基本的交通事故处理知识和财务核算能力。

**平台财务人员**：负责统筹费收缴确认、各子公司独立结算对账、退费审核与处理、三账户资金池管理、财务报表生成。财务人员需确保资金管理准确、透明。

**平台运营人员**：负责新企业入驻审核、企业信息维护、保障方案配置、费率规则维护、系统参数管理、通知模板维护等运营支撑工作。

**驾驶员（被保障驾驶员）**：本版中驾驶员不再是参保主体，而是被保障对象。驾驶员通过分司管理员导入后自动获得小程序账号（短信验证码登录），可进行事故申报、查询案件进度、查看个人驾驶评分。驾驶员不涉及缴费、车辆管理、退统等操作。

## 四、组织与数据架构

### 4.1 多租户架构

```
集团（Group）
  └─ 子公司A（SubCompany）[多租户 - 租户1]
  |   ├─ 车辆（Vehicle）
  |   |   ├─ 自有车辆（SELF_OWNED）── 公司资产
  |   |   └─ 挂靠车辆（AFFILIATED）── 个人车主挂靠
  |   |       └─ 统筹单（Policy）── 每车一份凭证
  |   |           └─ 案件（Case）── 关联凭证
  |   └─ 驾驶员（Driver）[池化管理 - 挂在集团下]
  |       └─ 驾驶员-车辆绑定（DriverVehicleBinding）── 白班/夜班
  |
  └─ 子公司B（SubCompany）[多租户 - 租户2]
  |   └─ ...（同上结构）
  |
  └─ 子公司C（SubCompany）[多租户 - 租户3]
      └─ ...（同上结构）

数据隔离边界：子公司之间完全隔离
数据共享边界：高管可查看本企业内跨分司统计数据（不能跨企业查看）
```

**架构设计原则**：

1. **子公司即租户**：每个子公司是一个独立的租户单元，拥有独立的数据空间、独立的结算账户、独立的业务数据。
2. **驾驶员池化共享**：驾驶员数据挂在集团级别，不隶属于某个子公司，可被不同子公司的分司管理员分配到各自车辆上。这解决了换班、借调等场景下驾驶员归属的灵活性问题。
3. **车辆归属明确**：每辆车辆严格归属于一个子公司，车辆产生的所有业务数据（凭证、案件、缴费记录）都在该租户范围内。
4. **凭证独立**：每辆车一份统筹单，凭证与车辆一对一绑定，一个凭证周期内只能有一个有效状态。

### 4.2 独立结算模式

各子公司与TSCS平台独立签约、独立缴费、独立对账、数据隔离。

**独立结算体系详细说明**：

**（1）合同管理**

- 每家子公司与TSCS平台单独签订《交通安全统筹合作协议》
- 合同内容包含：统筹方案、费率标准、缴费周期、双方权利义务、违约责任等
- 合同有效期通常为1年，到期前30天启动续约流程
- 每家子公司的合同独立存档，互不影响

**（2）缴费管理**

| 缴费场景 | 说明 | 操作方 |
|---------|------|--------|
| 首次参保 | 批量导入车辆后生成缴费单，企业确认后缴费 | 分司管理员 |
| 续保缴费 | 凭证到期前15天生成续保缴费单，企业确认后缴费 | 分司管理员 |
| 批量参保（追加） | 运营过程中新增车辆参保，生成缴费单后缴费 | 分司管理员 |
| 补缴 | 因欠费停保后恢复参保 | 分司管理员 |

- 缴费方式：对公转账（主要）、在线支付（辅助）
- 缴费周期：支持年缴、半年缴、季缴（按合同约定）
- 缴费确认：财务人员确认到账后系统自动激活凭证
- 欠费处理：逾期30天未缴费，系统发送欠费提醒；逾期60天未缴费，相关凭证自动进入"欠费停保"状态

**（3）发票管理**

- 每家子公司独立开具发票，发票抬头为该子公司全称
- 发票类型：增值税普通发票/增值税专用发票（按企业资质）
- 发票开具时机：缴费确认后，5个工作日内开具
- 发票内容：统筹服务费（注意：不可使用"统筹费用"等字样）
- 支持电子发票和纸质发票

**（4）续保管理**

- 系统自动检测凭证到期时间，提前30/15/7/1天发送多级提醒
- 分司管理员在到期前确认续保车辆清单
- 支持部分续保（剔除已报废或转出的车辆）
- 续保时费率按最新标准执行

**（5）退费管理**

- 退费申请由分司管理员发起
- 平台运营人员审核通过后计算退费金额
- 退费原路退回（退至原缴费账户）
- 自有车辆退费退给子公司对公账户
- 挂靠车辆退费退给实际缴费方（公司代缴退公司，车主自缴退车主）

### 4.3 自有车辆与挂靠车辆

#### 4.3.1 自有车辆与挂靠车辆对比表

| 维度 | 自有车辆（SELF_OWNED） | 挂靠车辆（AFFILIATED） |
|------|----------------------|----------------------|
| **资产归属** | 子公司资产，车辆登记在公司名下 | 个人车主资产，挂靠在子公司名下运营 |
| **产权证明** | 行驶证所有人为子公司 | 行驶证所有人为个人车主 |
| **缴费责任方** | 子公司统一缴纳 | 可由子公司代缴或车主自缴 |
| **退费受益方** | 退给子公司 | 退给实际缴费方 |
| **车辆管理权限** | 分司管理员全权管理 | 分司管理员管理运营信息，车主信息由平台维护 |
| **驾驶员绑定** | 分司管理员直接分配 | 分司管理员分配，但车主可能自行指定 |
| **参保审批** | 子公司管理员直接审批 | 需车主签署同意书后由分司管理员提交 |
| **出险给付** | 给付款支付至子公司指定账户 | 给付款支付协议约定（公司或车主） |
| **凭证管理** | 子公司统一管理 | 子公司统一管理，但车主可查看 |
| **报废/转出** | 分司管理员操作，凭证自动终止 | 需车主确认 + 分司管理员操作 |
| **发票** | 开具子公司发票 | 按实际缴费方开具对应发票 |
| **费用归属** | 子公司经营成本 | 视挂靠协议约定 |
| **参保意愿** | 强制参保（公司要求） | 需车主同意（合同约束或自愿） |

#### 4.3.3 挂靠车辆的特殊规则

**（1）参保规则**

- 挂靠车辆参保需车主签署《交通安全统筹参与同意书》（电子签名或纸质签署后上传）
- 分司管理员在系统中提交参保申请时，需上传车主同意书
- 系统验证车辆行驶证信息（OCR识别）与车主信息一致性
- 同一挂靠车辆同一时间只能参加一家子公司的统筹（不可重复参保）

**（2）缴费规则**

- 默认由子公司代缴（统一对公转账）
- 支持车主自缴模式（分司管理员生成个人缴费链接，车主通过H5页面完成缴费）
- 车主自缴的，发票可开给车主个人（普票）或子公司（专票）
- 缴费确认后，凭证立即生效

**（3）退费规则**

- 分司管理员发起退统申请
- 系统自动判断缴费方：
  - 公司代缴：退费至子公司对公账户
  - 车主自缴：退费至原缴费账户（个人）
- 部分退费场景（如车辆报废）：按实际已生效天数计算，扣除5%手续费
- 全额退费场景（如系统错误参保）：经平台运营审核后可全额退还（不扣手续费）

**（4）出险处理规则**

- 挂靠车辆出险时，互助给付款支付对象按统筹协议约定执行
- 若协议约定给付给子公司：给付款汇入子公司账户，由子公司与车主内部结算
- 若协议约定给付给车主：给付款汇入车主账户（需提供银行信息）
- 核查时，车主可参与（分司管理员协调）

### 4.4 驾驶员池化管理

#### 4.4.1 设计原则

驾驶员不隶属于任何子公司，而是挂在集团级别进行**池化管理**。这种设计源于运输行业的实际运营特点：出租车行业中，同一驾驶员可能在同一集团内不同子公司之间流转（换班、借调、转岗），将驾驶员绑定在某个子公司会导致管理僵化。

```
集团（驾驶员池）
  │
  ├─ 驾驶员A ─── 子公司A的车辆1（白班）
  ├─ 驾驶员B ─── 子公司A的车辆1（夜班）
  ├─ 驾驶员C ─── 子公司B的车辆5（白班）
  ├─ 驾驶员D ─── 未绑定（待分配）
  └─ ...
```

#### 4.4.2 驾驶员管理流程

**（1）驾驶员导入**

- 分司管理员通过Excel模板批量导入驾驶员信息（姓名、身份证号、手机号、准驾车型等）
- 系统自动创建驾驶员账号，归属到集团级驾驶员池
- 驾驶员首次通过小程序登录时，使用手机号 + 短信验证码认证
- 认证后自动绑定手机号对应的驾驶员账号

**（2）驾驶员分配到车辆**

- 分司管理员将驾驶员池中的驾驶员分配到本公司车辆上
- 分配时指定班次类型：白班 / 夜班 / 全天
- 一辆车可绑定多个驾驶员（通常白班+夜班各1人）
- 一个驾驶员可绑定多辆车（轮班场景，但同一时间段只能绑定1辆车）

**（3）白班/夜班绑定关系**

```
车辆A
  ├── 白班驾驶员：驾驶员X（06:00 - 18:00）
  └── 夜班驾驶员：驾驶员Y（18:00 - 06:00）
```

- 班次时间可由分司管理员自定义（不限于06:00-18:00）
- 系统记录班次调整历史，用于出险时确定当班驾驶员
- 出险时，系统根据出险时间自动匹配当班驾驶员

**（4）驾驶员离职处理**

- 分司管理员标记驾驶员为"已离职"状态
- 自动解除该驾驶员与所有车辆的绑定关系
- 驾驶员小程序账号保留，可查看历史数据（事故申报记录、驾驶评分等），但无法进行新操作
- 离职驾驶员的驾驶评分数据保留12个月，用于历史数据分析

**（5）驾驶员跨子公司调动**

- 原子公司分司管理员先解除绑定
- 新子公司分司管理员重新分配到本公司车辆
- 驾驶评分数据跟随驾驶员（不重置），确保评分连续性
- 历史出险记录随驾驶员保留

### 5.2 资金与账户原则

#### 5.2.1 三账户模型

子公司缴纳的每笔统筹费按以下比例分配至三个独立账户：

```
企业缴纳统筹费 100%
  ├── 互助互济资金池 ────────────── 70%
  │   用途：日常互助给付款支付
  │   特点：收入和支出高频流转
  │
  ├── 风险储备金 ────────────────── 15%
  │   用途：突发大额给付、年度结算补充
  │   特点：长期积累，非紧急不动用
  │
  └─ 平台运营管理费 ────────────── 15%
      用途：平台运营成本、人员薪酬、技术维护
      特点：平台收入来源
```

#### 5.2.2 各账户管理规则

**互助互济资金池（70%）**

- 按子公司独立记账，各子公司资金池互不混用
- 互助给付款从此账户支付
- 每月生成资金池余额报告
- 当某子公司资金池余额不足以支付待处理的互助给付款时，触发预警

**风险储备金（15%）**

- 按子公司独立记账
- 仅在以下情况可动用：
  - 单案给付金额超过该子公司资金池月均收入的3倍
  - 该子公司年度互助给付支出超过年度统筹费收入的80%
  - 经平台管理层审批同意的其他特殊情况
- 动用后需在下一季度内通过后续缴费逐步补充
- 风险储备金余额不低于该子公司当前在统总统筹费的10%

**平台运营管理费（15%）**

- 由TSCS平台统一管理
- 用于覆盖平台运营成本（人员、技术、办公等）
- 按子公司分别核算，便于合同续约时评估各子公司的服务成本

### 5.3 资金安全规则

**（1）风险储备金底线**

- 各子公司风险储备金余额不得低于当前在统总统筹费的10%
- 系统每日自动计算并检查风险储备金余额
- 当余额低于底线时，系统触发以下预警链：
  1. 向平台运营人员和财务人员发送预警通知
  2. 在平台端数据面板中高亮显示预警状态
  3. 向相关子公司分司管理员发送风险提示
  4. 自动暂停该子公司新增参保（已参保不受影响）

**（2）资金池不足处理逻辑**

当某子公司互助互济资金池余额不足以支付待处理的互助给付款时：

```
资金池不足
  │
  ├─ 不足金额 <= 风险储备金可用额度
  │   └─ 自动从风险储备金中调拨至资金池
  │       └─ 通知平台管理层和分司管理员
  │
  ├─ 不足金额 > 风险储备金可用额度
  │   └─ 触发紧急预警
  │       ├─ 通知平台总经理
  │       ├─ 通知分司管理员
  │       ├─ 暂停该子公司大额给付审批（>50,000元需总经理特批）
  │       └─ 建议分司管理员补充缴费或调整参保规模
  │
  └─ 资金池持续不足超过30天
      └─ 触发合同风险条款
          ├─ 平台运营人员与企业协商解决方案
          ├─ 可能方案：补充缴费、缩减保障范围、暂停统筹
          └─ 最坏情况：按合同约定终止统筹关系
```

**（3）资金安全底线**

- 严禁挪用互助互济资金池资金用于平台运营
- 严禁跨子公司调剂资金（各子公司独立结算）
- 严禁为平台运营提供担保或借贷
- 所有资金往来必须有完整的银行流水记录和系统记录

### 5.4 审批与留痕原则

**（1）全流程留痕**

- 所有业务操作必须记录完整的审计日志
- 审计日志内容包括：操作人、操作时间、操作类型、操作对象、操作前值、操作后值、IP地址、设备信息
- 审计日志不可修改、不可删除（只读）
- 审计日志保留期限：至少5年
- 关键业务节点必须有明确的责任人记录

**（2）审批链路记录**

- 分级审批的每级审批记录完整保留
- 审批意见、审批时间、审批人不可篡改
- 驳回时必须填写驳回原因
- 超时未审批时系统自动提醒（提醒间隔：24小时一次，最多提醒3次）

**（3）文档归档**

- 统筹单、统筹协议、互助理算单等核心单证电子化归档
- 归档文件采用PDF格式，不可修改
- 归档文件保存期限：凭证结束后至少10年
- 支持按子公司、时间范围、凭证号等维度检索

### 5.5 合规与隐私保护

**（1）敏感信息脱敏规则**

| 信息类型 | 脱敏规则 | 示例 |
|---------|---------|------|
| 身份证号 | 保留前3位和后4位 | 3201****1234 |
| 手机号 | 保留前3位和后4位 | 138****5678 |
| 银行账号 | 保留后4位 | ************1234 |
| 车架号（VIN） | 保留前3位和后4位 | LSG****5678 |
| 驾驶证号 | 同身份证号 | 3201****1234 |
| 车主姓名 | 姓保留，名用*替代 | 张** |

**（2）数据收集合规**

- 收集个人信息前需获得信息主体同意
- 通过统筹协议、统筹参与同意书等方式获得合法收集授权
- 仅收集与业务相关的必要信息，不超范围收集
- 提供信息查询、更正、删除渠道

**（3）账号注销**

- 驾驶员离职后，分司管理员标记离职状态，小程序账号冻结（可查不可操作）
- 企业注销后，所有相关分司管理员账号冻结
- 子公司可向平台申请正式注销，注销后：
  - 所有业务数据归档保留（满足法定保存期限）
  - 分司管理员账号永久禁用
  - 驾驶员账号恢复为未绑定状态

**（4）数据保留策略**

| 数据类型 | 保留期限 | 到期处理 |
|---------|---------|---------|
| 统筹单 | 凭证结束后10年 | 归档 |
| 案件记录 | 结案后10年 | 归档 |
| 财务记录 | 交易发生后10年 | 归档 |
| 操作日志 | 5年 | 归档后清理 |
| 驾驶评分 | 评分生成后2年 | 脱敏归档 |
| 车辆信息 | 车辆报废/转出后5年 | 脱敏归档 |
| 驾驶员信息 | 驾驶员注销后5年 | 脱敏归档 |
| 短信验证码 | 发送后5分钟 | 自动清理 |

### 5.6 消息触达原则

**（1）紧急消息 - 实时发送**

| 消息类型 | 接收方 | 触发条件 | 发送方式 |
|---------|--------|---------|---------|
| 新案件分配 | 核查员 | 案件分配到该核查员 | App推送 + 短信 |

**（2）重要消息 - 工作时间内发送**

| 消息类型 | 接收方 | 发送时间 | 发送方式 |
|---------|--------|---------|---------|
| 案件进度更新 | 驾驶员 | 工作时间（08:00-21:00） | 小程序推送 |
| 统筹单到期提醒 | 分司管理员 | 工作时间 | 企业端站内信 + 短信 |
| 缴费提醒 | 分司管理员 | 工作时间 | 企业端站内信 + 短信 |

**（3）非紧急消息 - 免扰策略**

| 消息类型 | 接收方 | 免扰时段 | 发送方式 |
|---------|--------|---------|---------|
| 安全评分更新 | 驾驶员 | 22:00-07:00 | 小程序站内信（次日推送） |
| 系统公告 | 全员 | 22:00-07:00 | 次日推送 |
| 安全培训通知 | 驾驶员 | 22:00-07:00 | 次日推送 |

**（4）消息频率限制**

- 同一类型消息对同一接收方，24小时内最多发送3条
- 紧急消息不受频率限制
- 用户可在小程序中设置免扰时段

### 6.2 企业入驻与签约流程

**参与方**：企业代表、平台运营人员、系统

**流程说明**：

```
企业代表                 系统                  平台运营人员
  │                      │                        │
  ├─ 1.提交入驻申请 ────>│                        │
  │                      ├─ 2.生成入驻工单 ──────>│
  │                      │                        ├─ 3.审核企业资质
  │                      │                        │  (营业执照/道路运输经营许可证/
  │                      │                        │   安全生产许可证/法人身份证)
  │                      │                        │
  │                      │<─ 4.提交审核结果 ──────┤
  │<─ 5.通知审核结果 ────┤                        │
  │                      │                        │
  ├─ 6.确认合同条款 ────>│                        │
  │                      ├─ 7.生成合同草案 ──────>│
  │                      │                        ├─ 8.确认并签署合同
  │                      │                        │  (电子签/纸质签)
  │                      │                        │
  │                      │<─ 9.上传合同文件 ──────┤
  │                      ├─ 10.开通企业端账号      │
  │<─ 11.发送账号信息 ───┤   (账号+初始密码)      │
  │                      │                        │
  ├─ 12.首次登录设置 ───>│                        │
  │   (修改密码/补充信息)  │                        │
  │                      │                        │
```

**详细步骤**：

**步骤1：提交入驻申请**
- 企业代表通过TSCS平台入驻页面（H5或企业端注册入口）填写入驻申请表
- 必填信息：企业全称、统一社会信用代码、企业地址、联系人姓名、联系电话、电子邮箱
- 可选信息：企业规模（车辆数量、驾驶员数量）、业务类型（出租/租赁/客运/旅游/货运）
- 上传材料：营业执照扫描件、道路运输经营许可证扫描件（如有）、安全生产许可证（如有）、法人身份证正反面

**步骤2：生成入驻工单**
- 系统自动创建入驻审核工单，分配工单编号
- 工单状态：待审核
- 通知平台运营人员有新的入驻申请待处理

**步骤3：审核企业资质**
- 平台运营人员登录平台端，查看入驻工单详情
- 逐项核查企业提交的资质材料：
  - 营业执照：验证真伪、确认经营范围包含相关运输业务
  - 道路运输经营许可证：确认运输资质合法有效
  - 安全生产许可证：确认安全管理资质（如适用）
  - 法人身份证：与营业执照法人信息一致
- 可进行工商信息核验（对接国家企业信用信息公示系统）

**步骤4：提交审核结果**
- 平台运营人员在工单中填写审核意见
- 审核结果：通过 / 驳回（需填写驳回原因）
- 驳回原因类型：材料不完整、材料不合规、经营范围不符、信用异常、其他

**步骤5：通知审核结果**
- 系统通过短信和邮件通知企业代表审核结果
- 驳回的，附带详细驳回原因和修改建议
- 企业可修改后重新提交（最多3次）

**步骤6：确认合同条款**
- 审核通过后，系统生成合同条款预览页面
- 企业代表在线确认合同条款（不可修改）
- 合同核心条款包括：统筹方案说明、费率标准说明、缴费周期、双方权利义务、违约责任、争议解决方式等

**步骤7：生成合同草案**
- 系统根据合同模板和企业信息生成合同草案
- 合同编号规则：TSCS-CONTRACT-{子公司编码}-{年份}-{序号}
- 合同附件：保障方案明细表、费率表、非保险告知书

**步骤8：签署合同**
- 支持电子签名（推荐）或纸质签署
- 电子签名：通过第三方电子签平台完成，具有法律效力
- 纸质签署：双方各自盖章签字后上传扫描件
- 合同生效日：签署日或合同约定生效日

**步骤9：上传合同文件**
- 签署完成后，合同文件归档至系统
- 合同文件包括：合同正文、附件、签署页
- 归档格式：PDF（不可修改）
- 合同到期前30天系统自动提醒续约

**步骤10：开通企业端账号**
- 系统自动创建企业端账号，初始角色为分司管理员
- 账号规则：管理员手机号作为登录账号
- 生成初始密码并通过短信发送
- 同时创建该子公司的租户数据空间
- 企业可根据需要由平台运营人员增设安全员和高管角色账号

**步骤11：发送账号信息**
- 通过短信向分司管理员发送：登录地址、账号、初始密码（安全员和高管账号由企业内部另行分配）
- 同时通过邮件发送：《企业端使用手册》、《安全管理指南》
- 账号有效期为收到通知后30天，逾期未激活需联系平台重新激活

**步骤12：首次登录设置**
- 分司管理员/安全员/高管首次登录需修改初始密码
- 补充企业详细信息：对公账户信息、开票信息、紧急联系人、组织架构等
- 设置安全选项：二次验证（可选）、操作密码（可选）
- 完成新手引导：了解各功能模块位置和操作方式

### 6.3 批量参保流程（核心流程）

**参与方**：分司管理员、系统、平台运营人员

**流程说明**：

```
分司管理员               系统                    平台运营人员
  │                      │                        │
  ├─ 1.下载导入模板 ────>│                        │
  │<─ 返回Excel模板 ─────┤                        │
  │                      │                        │
  ├─ 2.填写并上传 ──────>│                        │
  │                      ├─ 3.数据校验             │
  │                      │  (格式/重复/基础规则)    │
  │<─ 返回校验结果 ──────┤                        │
  │   (通过/失败明细)     │                        │
  │                      │                        │
  ├─ 4.确认导入 ────────>│                        │
  │   (仅导入通过项)      │                        │
  │                      ├─ 5.创建车辆记录         │
  │                      ├─ 6.关联保障方案         │
  │                      ├─ 7.生成统筹单(待缴费)  │
  │                      │                        │
  │<─ 返回参保确认单 ────┤                        │
  │   (车辆清单+费用明细)  │                        │
  │                      │                        │
  ├─ 8.确认参保 ────────>│                        │
  │                      ├─ 9.生成缴费单            │
  │<─ 返回缴费单 ────────┤                        │
  │                      │                        │
  ├─ 10.完成缴费 ───────>│                        │
  │                      ├─ 11.确认到账             │
  │                      ├─ 12.激活凭证             │
  │                      ├─ 13.绑定驾驶员(可选)       │
  │<─ 返回参保成功通知 ──┤                        │
  │                      │                        │
```

**详细步骤**：

**步骤1：下载导入模板**
- 分司管理员在企业端"车辆管理"模块下载批量参保导入模板
- 模板为标准化Excel格式，包含以下字段：
  - 必填：车牌号、车牌颜色、车型、品牌型号、车架号、发动机号、注册日期、使用性质、车辆归属（自有/挂靠）、保障方案选择
  - 可选：座位数、排量、燃料类型、车主姓名（挂靠车辆必填）、车主身份证号（挂靠车辆必填）、车主手机号（挂靠车辆必填）
- 模板中包含填写说明和示例数据

**步骤2：填写并上传**
- 分司管理员按模板格式填写车辆信息
- 单次导入上限：1000条记录
- 超过1000条需分批导入
- 上传方式：拖拽上传或点击选择文件
- 支持格式：.xlsx、.xls

**步骤3：数据校验**
- 系统对上传数据进行多维度校验：
  - 格式校验：必填字段是否为空、格式是否正确（车牌号格式、VIN格式、日期格式等）
  - 重复校验：车牌号、VIN号是否在本企业内已存在
  - 跨企业重复校验：车牌号、VIN号是否在其他企业已参保（防止重复参保）
  - 规则校验：注册日期是否合理（不超过当前日期）、使用性质是否在可选范围内
  - 挂靠车辆特殊校验：车主姓名和身份证号是否匹配
- 校验结果分为：通过、警告（可修复）、错误（不可导入）
- 返回详细的校验结果报告（每个字段的校验状态）

**步骤4：确认导入**
- 分司管理员查看校验结果，可选择：
  - 全部导入（仅导入通过项）
  - 修复后重新上传（针对警告/错误项）
  - 取消导入
- 确认导入后，系统进入数据处理阶段

**步骤5：创建车辆记录**
- 系统为每条通过的记录创建车辆数据
- 自动分配车辆ID
- 车辆初始状态：ACTIVE
- 参保状态：PENDING（待参保）
- 记录创建人和创建时间

**步骤6：关联保障方案**
- 系统根据导入时选择的保障方案为每辆车关联方案
- 保障方案包含：保障责任范围、给付限额、免赔设定等
- 如导入时未指定方案，使用该企业默认保障方案

**步骤7：生成统筹单（待缴费状态）**
- 系统为每辆车生成一份统筹单
- 凭证状态：PENDING_PAYMENT（待缴费）
- 凭证编号规则：TSCS-{子公司编码}-{年份}-{月份}-{序号}
- 凭证有效期：默认1年（可配置为半年、季度）
- 生成费用明细：每辆车的统筹费计算结果

**步骤8：返回参保确认单**
- 系统生成参保确认单，包含：
  - 车辆清单（车牌号、车型、保障方案、统筹费）
  - 费用汇总（车辆数、总统筹费、规模折扣、应付金额）
  - 企业系数和费率明细
  - 缴费截止日期（默认15个工作日）
- 分司管理员确认后进入缴费环节

**步骤9：生成缴费单**
- 系统根据参保确认单生成正式缴费单
- 缴费单编号：TSCS-PAY-{子公司编码}-{年份}-{月份}-{序号}
- 缴费单包含：缴费金额、收款账户信息、缴费截止日期
- 收款信息：TSCS平台对公账户（开户行、户名、账号）
- 缴费方式说明：对公转账（附言填写缴费单编号）

**步骤10：完成缴费**
- 分司管理员通过企业对公账户完成转账
- 转账附言填写缴费单编号，便于平台识别
- 分司管理员在企业端上传银行转账凭证
- 如有需要，可联系平台财务确认收款信息

**步骤11：确认到账**
- 平台财务人员在平台端确认收款
- 系统自动匹配缴费单和转账记录（通过缴费单编号）
- 确认到账后更新缴费单状态为已缴费

**步骤12：激活凭证**
- 缴费确认后，系统自动激活所有相关统筹单
- 凭证状态从 PENDING_PAYMENT 变更为 ACTIVE
- 凭证生效日期：缴费确认日或合同约定生效日（取较晚者）
- 通知分司管理员参保成功（站内信 + 短信）

**步骤13：绑定驾驶员（可选）**
- 分司管理员在参保完成后绑定驾驶员到车辆
- 可在车辆详情页进行单辆绑定，也可批量绑定
- 绑定方式：从驾驶员池中选择驾驶员，指定班次（白班/夜班/全天）
- 驾驶员绑定后可通过小程序查看自己名下的统筹单

**挂靠车辆特殊处理**：

在步骤4确认导入前，挂靠车辆还需：
- 上传车主签署的《交通安全统筹参与同意书》
- 若车主选择自缴，系统生成个人缴费链接（H5页面）
- 车主自缴完成后，该车辆凭证方可激活
- 部分车主自缴、部分公司代缴的场景下，可分批激活

### 6.4 事故申报流程

**参与方**：驾驶员、系统、安全员、调度员、核查员

**申报渠道**：
- **驾驶员端小程序**（常规渠道，唯一常规申报入口）：驾驶员第一时间通过小程序提交事故信息，系统同时推送到企业端和平台端
- **电话申报**（异常渠道）：驾驶员通过电话向平台申报，由调度员手动录入系统。电话申报仅作为小程序无法使用时的备用渠道

> **注**：不再设置"分司管理员代申报"和"平台调度员代申报"作为常规渠道。所有事故由驾驶员直接申报，企业端安全员负责审核，平台端调度员负责后续处理。

**流程说明**：

```
驾驶员              系统              安全员            调度员           核查员
  │                 │                 │                │                │
  ├─ 1.发起事故申报 ─>│                │                │                │
  │  (选择车辆/      ├─ 2.生成申报号     │                │                │
  │   拍照/描述)     ├─ 3.关联车辆信息   │                │                │
  │                 ├─ 4.匹配当班驾驶员   │                │                │
  │                 │                 │                │                │
  ├─ 5.同步推送到 ─>├────────────────>│                │                │
  │   企业端+平台端  │                 │                │                │
  │                 │                 │                │                │
  │                 │                 ├─ 6.企业审核     │                │
  │                 │                 │  (必经环节)      │                │
  │                 │                 │                │                │
  │                 │                 ├─ 7.审核操作     │                │
  │                 │                 │  通过/驳回      │                │
  │                 │                 │                 │                │
  │                 │  (审核通过)       │                 │                │
  │                 │<────────────────┤                 │                │
  │                 ├─ 8.案件分类 ─────────────────────>│                │
  │                 │                 │                │                │
  │                 │                 │                ├─ 9.案件分配    │
  │                 │                 │                │  (分配核查员)   │
  │                 │                 │                │                │
  │                 │                 │                ├─ 10.受理通知 ─>│
  │                 │                 │                │<───────────────┤
  │<─ 11.受理通知 ──┤                 │                │  核查员确认接收  │
  │                 │                 │                │                │
```

**企业端审核为必经环节**：所有事故申报必须经过企业端安全员审核后才能流转到平台端处理。

**超时自动上报规则**：
- 企业审核超时时间可配置（默认4个工作小时）
- 超时未审核 → 系统自动提醒安全员（企业端站内信 + 短信）
- 超时2倍（8个工作小时）仍未审核 → 系统自动将案件上报至平台端（避免卡住），同时抄送平台运营人员

**详细步骤**：

**步骤1：发起事故申报**
- 驾驶员登录小程序后点击"一键事故申报"
- 选择出险车辆（仅显示当前绑定的车辆）
- 填写出险信息：
  - 出险时间（默认当前时间，可修改）
  - 出险地点（手动输入，不自动GPS定位）
  - 事故类型：单方事故 / 双方事故 / 多方事故 / 其他
  - 事故描述（文本，最多500字）
  - 是否有人伤：是/否（如选是，需填写伤情描述）
  - 是否报交警：是/否
- 拍照上传（最多9张）：
  - 事故全景照（1张，必拍）
  - 车辆受损部位特写（至少1张）
  - 对方车辆受损情况（如涉及，至少1张）
  - 现场环境照（可选）
- 提交事故申报

**步骤2：生成申报号**
- 系统自动生成唯一申报号
- 申报号规则：TSCS-RPT-{年份}-{月份}-{日}-{序号}
- 案件状态：REPORTED（已申报）

**步骤3：关联车辆信息**
- 系统根据车牌号自动关联车辆信息
- 提取：车辆所属子公司、保障方案、凭证信息、绑定驾驶员、参保状态
- 校验参保状态：
  - ACTIVE（在统）：正常受理
  - EXPIRED（已过期）：标记为非统筹案件，但仍受理（通知分司管理员需续保）
  - SUSPENDED（欠费停保）：标记为受限受理
  - NONE（未参统）：引导到企业端参统流程

**步骤4：匹配当班驾驶员**
- 系统根据出险时间自动匹配当班驾驶员
- 匹配规则：查找该车辆在出险时间段内的绑定驾驶员（白班/夜班）
- 若申报人不是当班驾驶员，系统标记差异，提醒安全员核实
- 若车辆未绑定驾驶员，提示分司管理员补充驾驶员信息

**步骤5：同步推送到企业端和平台端**
- 驾驶员提交事故申报后，系统同时将申报信息推送到：
  - **企业端**：进入安全员的审核待办列表
  - **平台端**：进入调度员的待受理列表（但处于待审核状态，企业审核通过后才可操作）

**步骤6：企业端审核（必经环节）**

安全员负责审核事故申报信息的三个核心要素：

| 审核要素 | 审核内容 | 不通过处理 |
|---------|---------|-----------|
| **完整性** | 照片是否齐全（事故全景照、受损部位特写）、出险时间/地点/事故描述等必填项是否完整 | 驳回，要求驾驶员补充材料 |
| **真实性** | 是否属于参统车辆、是否在参统期间、是否为当班驾驶员 | 驳回，标记疑点并说明原因 |
| **合规性** | 事故类型是否符合统筹范围（酒驾/无证驾驶/故意行为等除外） | 驳回，说明不合规原因 |

- **审核人**：安全员（由企业在系统配置中指定）
  - 支持单级审核（1人）或多级审核（如：安全员 → 车队经理）
  - 多级审核时，每一级通过后才流转到下一级
- **免审规则**（系统自动执行）：
  - 紧急案件（人伤 / 特急）自动跳过企业审核，直接上报平台
  - 超时自动上报（超过配置的超时时间，系统自动上报平台）
- **审核操作**：
  - 审核通过 → 案件流转到平台调度员受理（案件状态：REPORTED → PENDING_DISPATCH）
  - 审核驳回 → 退回驾驶员补充材料，需填写驳回原因（案件状态：REPORTED → REJECTED_BY_ENTERPRISE）
  - 驾驶员修改后可重新提交，再次进入审核流程

**步骤7：平台端接手后的流程**

企业审核通过（或超时自动上报）后，案件进入平台端处理：

1. **案件分类**：系统自动分类 + 调度员人工确认/调整
   - 案件性质：单方事故 / 双方事故 / 多方事故 / 人伤事故 / 货损事故 / 其他
   - 严重程度：轻微（预估损失<=5,000） / 一般（5,000<预估<=30,000） / 重大（>30,000） / 特大（涉及人员重伤或死亡）
   - 紧急程度：普通 / 紧急（人伤/重大财产损失）/ 特急（多人伤亡）

2. **案件分配**：调度员手动分配核查员或系统自动分配
   - 优先分配给距离出险地点最近的空闲核查员
   - 考虑核查员当前任务量（避免超载）
   - 考虑核查员的专业能力（人伤案件优先分配有经验的核查员）
   - 分配后案件状态：ASSIGNED（已分配）

3. **受理通知**：系统通过App推送通知核查员，核查员在App中确认接收任务

**步骤8~11：后续流程**

审核通过后，平台端按以下流程依次处理：
- 调度任务 → 事故核查（现场核查） → 损失评估 → 生成损失核定单 → 互助理算 → 审批 → 互助给付 → 结案

（具体流程详见6.5核查损失评估流程、6.6互助给付流程）

### 6.5 核查损失评估流程

**参与方**：核查员、系统、审批人、修理厂

**流程说明**：

```
核查员                 系统                审批人           修理厂
  │                   │                   │                │
  ├─ 1.接收任务 ─────>│                   │                │
  │<─ 任务详情 ───────┤                   │                │
  │                   │                   │                │
  ├─ 2.出发前往现场    │                   │                │
  ├─ 3.到达签到 ─────>│                   │                │
  │   (手动签到+地址)  ├─ 记录到达时间      │                │
  │                   │                   │                │
  ├─ 4.现场拍照取证 ─>│                   │                │
  │   (至少5张)        ├─ 存储照片          │                │
  │                   │                   │                │
  ├─ 5.当事人笔录 ───>│                   │                │
  │   (询问+记录)      ├─ 存储笔录          │                │
  │                   │                   │                │
  ├─ 6.初步定责 ─────>│                   │                │
  │   (全责/主责/      │                   │                │
  │    同责/次责/无责)  │                   │                │
  │                   │                   │                │
  ├─ 7.损失评估 ─────>│                   │                │
  │   (逐项评估)       ├─ 生成评估单        │                │
  │                   │                   │                │
  │                   ├─ 8.分级审批 ──────>│                │
  │                   │  (按金额自动判断)    │                │
  │                   │                   │                │
  │                   │  <=5000: 自动通过    │                │
  │                   │  <=30000: 主管审批   │                │
  │                   │  >30000: 经理审批    │                │
  │                   │                   │                │
  │                   │<─ 9.审批结果 ───────┤                │
  │                   │                   │                │
  │                   ├─ 10.修理厂报价 ────┤───────────────>│
  │                   │<─ 返回维修报价 ─────┤───────────────┤
  │                   │                   │                │
  │<─ 核查完成通知 ───┤                   │                │
```

**详细步骤**：

**步骤1：接收任务**
- 核查员在现场端App中查看待处理任务列表
- 点击任务查看详情：申报号、出险地点、事故类型、申报人信息、车辆信息、保障方案
- 核查员确认接受任务或拒绝（拒绝需说明原因）
- 案件状态：DISPATCHED（已派勘）

**步骤2：出发前往现场**
- 核查员确认出发，系统记录出发时间
- 核查员可通过App导航到出险地点
- 若预计无法在合理时间内到达（如距离过远），核查员可申请更换核查员

**步骤3：到达签到**
- 核查员到达现场后，在App中点击"到达签到"
- 系统记录到达时间（取App提交时间）
- 核查员输入/确认现场地址（不使用GPS自动定位）
- 签到超时提醒：若任务分配后超过2小时未签到，系统提醒调度员跟进
- 案件状态：ON_SITE（现场核查中）

**步骤4：现场拍照取证**
- 核查员使用App内置相机拍照
- 必拍照片（最少5张，最多20张）：
  1. 事故全景照（含车辆和路面环境，至少1张）
  2. 涉事车辆全景照（每辆车至少1张，正侧面45度角）
  3. 受损部位特写（每个受损部位至少1张，含参照物）
  4. 对方车辆受损部位（如涉及双方/多方事故）
  5. 现场道路环境（路面状况、交通标志等）
  6. 交警责任认定书（如已出具）
- 拍照要求：
  - 照片需包含时间水印
  - 受损部位特写需包含参照物（如硬币、尺子）用于判断损伤程度
  - 不可使用滤镜或美颜功能

**步骤5：当事人笔录**
- 核查员在App中填写当事人笔录表
- 笔录内容包括：
  - 驾驶员姓名、驾驶证号、联系方式
  - 事故经过描述（核查员记录或当事人自述）
  - 当事人初步陈述的责任意见
  - 是否有伤者（如有，记录伤者信息和伤情）
  - 对方信息（如涉及）：驾驶员、车牌号、保险公司/统筹平台
  - 交警处理情况（是否已到场、是否出具认定书）
  - 目击者信息（如有）
- 笔录支持语音转文字功能（核查员口述，系统自动转录）
- 笔录完成后需当事人电子签名确认

**步骤6：初步定责**
- 核查员根据现场情况和当事人陈述进行初步定责
- 定责选项：全部责任 / 主要责任 / 同等责任 / 次要责任 / 无责任 / 待定（等交警认定书）
- 若有交警责任认定书，以认定书为准
- 初步定责结果仅用于后续补偿参考，最终以综合审核结果为准

**步骤7：损失评估**
- 核查员在现场对车辆损失进行逐项评估
- 评估内容：
  - 损失部位（如：前保险杠、左前翼子板、大灯等）
  - 损失类型（钣金修复/更换零件/喷漆/其他）
  - 预估损失金额（参考系统内置的维修配件价格库）
  - 是否需要拖车（是/否，拖车费用另计）
- 系统生成损失评估单（含逐项明细）
- 损失评估单状态：DRAFT（草稿）

**步骤8：分级审批**
- 核查员提交损失评估单后，系统按预估总损失金额自动进入对应审批层级：

| 预估损失金额 | 审批层级 | 审批人 | 审批时效 |
|------------|---------|--------|---------|
| <= 5,000元 | 自动通过 | 系统 | 即时 |
| 5,001 ~ 30,000元 | 主管审批 | 核查主管 | 4个工作小时 |
| > 30,000元 | 经理审批 | 业务经理 | 8个工作小时 |

- 审批人可在评估单上批注意见
- 驳回时需填写驳回原因，核查员修改后重新提交
- 超时未审批自动升级至上一级

**步骤9：审批结果**
- 审批通过后，损失评估单状态变为 APPROVED（已审批）
- 系统通知核查员、分司管理员审批结果
- 案件状态：ASSESSED（已评估）

**步骤10：修理厂报价**
- 若车辆需要维修，核查员或分司管理员可推荐修理厂
- 修理厂在系统中提交维修报价单（或核查员代为录入）
- 报价单包含：维修项目、配件清单、工时费、总计金额
- 系统对比评估金额与报价金额：
  - 报价 <= 评估金额 * 1.1：直接通过
  - 报价 > 评估金额 * 1.1：需审批人确认
- 最终维修费用以实际报价为准（经审批后）

**核查完成后续动作**：
- 核查员完成所有现场工作后，在App中提交核查报告
- 系统汇总所有现场数据（照片、笔录、评估单、审批记录）
- 通知分司管理员核查已完成，案件进入材料审核和互助给付阶段

### 6.6 互助给付流程

**参与方**：系统、互助给付员、分司管理员/高管、审批人、财务人员

**流程说明**：

```
分司管理员            系统              互助给付员         审批人         财务
  │                  │                   │               │             │
  ├─ 1.上传给付材料 ─>│                   │               │             │
  │  (维修发票/       │                   │               │             │
  │   认定书/清单等)   │                   │               │             │
  │                  ├─ 2.材料完整性校验   │               │             │
  │                  │                   │               │             │
  │                  ├─ 3.材料审核 ──────>│               │             │
  │                  │                   ├─ 4.逐项审核    │             │
  │                  │                   │  (发票真伪/     │             │
  │                  │                   │   金额一致/     │             │
  │                  │                   │   材料齐全)     │             │
  │                  │                   │               │             │
  │                  │                   ├─ 5.互助理算    │             │
  │                  │                   │  (公式计算)     │             │
  │                  │                   │               │             │
  │                  │                   ├─ 生成计算书 ──>│             │
  │                  │                   │               │             │
  │                  │                   ├─ 6.分级审批 ──>│             │
  │                  │                   │               │             │
  │                  │  <=2000: 自动通过    │               │             │
  │                  │  <=50000: 互助给付员│               │             │
  │                  │           +企业确认  │               │             │
  │                  │  <=100000: 经理+企业 │               │             │
  │                  │  >100000: 总经理+企业│               │             │
  │                  │                   │               │             │
  │<─ 7.企业确认请求 ─┤                   │               │             │
  ├─ 企业确认 ───────>│                   │               │             │
  │                  │                   │               │             │
  │                  │<─ 8.审批结果 ───────┤───────────────┤             │
  │                  │                   │               │             │
  │                  ├─ 9.生成支付指令 ────────────────────────────────>│
  │                  │                   │               │             │
  │                  │                   │               │             ├─ 10.执行支付
  │                  │                   │               │             │
  │                  │<─ 11.支付完成 ─────────────────────────────────┤
  │<─ 给付到账通知 ───┤                   │               │             │
```

**详细步骤**：

**步骤1：上传给付材料**
- 分司管理员（或驾驶员协助）在案件详情页上传互助给付所需材料
- 必要材料清单：
  - 交通事故责任认定书（原件扫描件）
  - 维修发票（原件扫描件）
  - 维修清单/报价单（修理厂盖章）
  - 拖车费发票（如有拖车）
  - 医疗费用发票（如有人伤）
  - 伤残鉴定报告（如有人伤致残）
  - 其他证明材料（如施救费、停车费等）
- 材料上传支持：拍照上传、文件上传（PDF/JPG/PNG）
- 系统检查材料类型是否齐全，缺失项明确提示

**步骤2：材料完整性校验**
- 系统根据案件类型和保障方案自动校验必要材料是否齐全
- 校验规则按案件类型区分：
  - 财损案件：认定书 + 维修发票 + 维修清单
  - 人伤案件：认定书 + 医疗发票 + 诊断证明 + 伤残报告（如适用）
  - 货损案件：认定书 + 货物损失清单 + 价值证明
- 校验结果：完整 / 不完整（列出缺失项）

**步骤3：材料审核**
- 互助给付员在平台端查看待审核案件列表
- 逐项审核材料：

| 审核项 | 审核内容 | 不通过处理 |
|--------|---------|-----------|
| 责任认定书 | 真伪校验、事故信息一致性 | 要求提供原件或联系交警核实 |
| 维修发票 | 发票真伪（税务系统校验）、金额合理性 | 退回重新提供 |
| 维修清单 | 维修项目与损失评估是否一致 | 标记差异项，要求说明 |
| 医疗发票 | 发票真伪、费用合理性 | 退回重新提供 |
| 时间一致性 | 出险时间、维修时间、发票时间逻辑一致 | 标记异常，要求说明 |
| 金额合理性 | 维修费用与市场均价对比 | 标记偏高项，要求说明或补充报价 |

**步骤4：逐项审核详情**
- 互助给付员对每份材料进行审核并记录审核意见
- 审核结果选项：通过 / 需补充材料 / 驳回
- 需补充材料：列出缺失材料清单，系统通知分司管理员补充
- 驳回：填写驳回原因，案件退回分司管理员

**步骤5：互助理算**
- 材料审核通过后，互助给付员发起互助理算
- 互助理算公式：

```
互助给付款 = (损失额 x 责任比例 - 免赔额) x (1 - 协议扣除比例) - 已付金额
```

各参数说明：

| 参数 | 说明 | 取值规则 |
|------|------|---------|
| 损失额 | 经审核确认的实际损失金额 | 取维修发票金额与评估金额的较低者 |
| 责任比例 | 依据责任认定书确定的责任比例 | 全责100% / 主责70% / 同责50% / 次责30% / 无责0% |
| 免赔额 | 按保障方案约定的免赔额 | 标准方案：500元/次 |
| 协议扣除比例 | 按统筹协议约定的扣除比例 | 默认0%（无扣除） |
| 已付金额 | 之前已垫付的金额 | 如平台已预付施救费等 |

- 计算示例：
  - 损失额：8,000元
  - 责任比例：主要责任（70%）
  - 免赔额：500元
  - 协议扣除比例：0%
  - 已付金额：0元
  - 互助给付款 = (8,000 x 70% - 500) x (1 - 0%) - 0 = 5,100元

- 互助给付员可调整互助理算参数（需填写调整理由）
- 系统生成《互助理算单》，包含完整的计算过程

**步骤6：分级审批**
- 互助理算单生成后，系统根据给付金额自动进入对应审批层级：

| 互助给付金额 | 审批层级 | 审批人 | 企业确认 | 审批时效 |
|------------|---------|--------|---------|---------|
| <= 2,000元 | 自动通过 | 系统 | 不需要 | 即时 |
| 2,001 ~ 50,000元 | 互助给付员 + 企业确认 | 互助给付员审核 | 需要 | 2个工作日 |
| 50,001 ~ 100,000元 | 经理 + 企业确认 | 业务经理 | 需要 | 3个工作日 |
| > 100,000元 | 总经理 + 企业确认 | 平台总经理 | 需要 | 5个工作日 |

**步骤7：企业确认请求**
- 中额以上（>2,000元）补偿需要高管确认
- 系统向分司管理员发送确认请求（企业端站内信 + 短信）
- 分司管理员在企业端查看给付计算书详情：
  - 损失明细
  - 责任比例
  - 计算过程
  - 最终给付金额
- 分司管理员可选择：确认 / 有异议
- 选择"有异议"时，需填写异议内容和期望金额，系统转回互助给付员复核

**步骤8：审批结果**
- 所有审批环节通过后，系统记录最终审批结果
- 审批结果：通过 / 驳回（需填写原因）
- 通过后，案件状态：APPROVED_FOR_PAYMENT（已审批待支付）

**步骤9：生成支付指令**
- 系统根据审批通过的给付金额生成支付指令
- 支付指令包含：
  - 案件编号
  - 收款方信息（自有车辆=子公司账户，挂靠车辆=协议约定账户）
  - 支付金额
  - 支付方式（银行转账）
  - 付款备注（TSCS互助给付-案件号）
- 支付指令提交至平台财务人员

**步骤10：执行支付**
- 平台财务人员在平台端审核支付指令
- 核对收款账户信息、支付金额
- 通过企业网银执行转账
- 在系统中记录支付信息：转账流水号、转账时间、实际支付金额

**步骤11：支付完成**
- 财务人员确认支付完成后，系统更新案件状态
- 案件状态：PAID（已支付）
- 通知分司管理员和驾驶员给付款已支付
- 分司管理员可在企业端下载《互助理算单》和支付凭证
- 案件进入结案流程

### 6.7 退统退费流程

**参与方**：分司管理员/高管（退统审批）、系统、平台运营人员、平台财务人员

**流程说明**：

```
分司管理员              系统                平台运营人员         平台财务
  │                    │                    │                 │
  ├─ 1.发起退统申请 ──>│                    │                 │
  │  (选择车辆/原因)    │                    │                 │
  │                    ├─ 2.退费计算         │                 │
  │                    │  (公式计算)         │                 │
  │                    │                    │                 │
  │<─ 返回退费明细 ────┤                    │                 │
  │                    │                    │                 │
  ├─ 3.确认退统 ──────>│                    │                 │
  │                    ├─ 4.提交审核 ──────>│                 │
  │                    │                    ├─ 5.审核申请     │
  │                    │                    │  (合规性/材料)   │
  │                    │                    │                 │
  │                    │<─ 6.审核结果 ───────┤                 │
  │<─ 通知审核结果 ────┤                    │                 │
  │                    │                    │                 │
  │                    │  (审核通过)         │                 │
  │                    ├─ 7.生成退费指令 ──────────────────────>│
  │                    │                    │                 ├─ 8.执行退费
  │                    │                    │                 │
  │                    │<─ 9.退费完成 ─────────────────────────┤
  │<─ 退统完成通知 ────┤                    │                 │
  │                    ├─ 10.凭证作废        │                 │
  │                    ├─ 11.记录退统历史     │                 │
```

**详细步骤**：

**步骤1：发起退统申请**
- 分司管理员在企业端"车辆管理"中选择需要退统的车辆
- 支持单辆退统和批量退统
- 填写退统原因：
  - 车辆报废
  - 车辆转让/过户
  - 企业注销/终止运营
  - 保障方案调整
  - 其他（需说明原因）
- 上传支持材料（根据退统原因）：
  - 车辆报废：报废证明
  - 车辆转让：过户证明
  - 企业注销：注销证明
  - 其他：情况说明

**步骤2：退费计算**
- 系统自动计算退费金额
- 退费公式：

```
应退统筹费 = (统筹费总额 - 已生效天数 x 日均费) x (1 - 5%手续费)
```

各参数说明：
- 统筹费总额：该车辆实际缴纳的统筹费总额
- 已生效天数：从凭证生效日到退统申请日的天数
- 日均费：统筹费总额 / 凭证总天数
- 手续费：固定5%（最低0元）

- 退费计算示例：
  - 统筹费总额：3,600元
  - 凭证总天数：365天
  - 已生效天数：120天
  - 日均费：3,600 / 365 = 9.86元/天
  - 未生效天数价值：(365 - 120) x 9.86 = 2,415.70元
  - 手续费：2,415.70 x 5% = 120.79元
  - 应退统筹费：2,415.70 - 120.79 = 2,294.91元

- 特殊情况：
  - 凭证生效不足7天：全额退还（不扣手续费）
  - 凭证生效不足30天：退还90%（不另扣手续费）
  - 凭证生效超过30天：按上述公式计算
  - 有未结案件：需先结案或明确案件处理方案后才能退统

**步骤3：确认退统**
- 系统展示退费计算明细：
  - 统筹费总额
  - 已生效天数和对应费用
  - 未生效天数和对应费用
  - 手续费
  - 应退金额
  - 退费账户信息（自有车辆=子公司账户，挂靠车辆=实际缴费方账户）
- 分司管理员确认退统信息后提交审核

**步骤4：提交审核**
- 系统生成退统审核工单
- 工单包含：退统车辆清单、退费明细、支持材料
- 通知平台运营人员有新的退统申请待审核

**步骤5：审核申请**
- 平台运营人员逐项审核：
  - 退统原因是否合规
  - 支持材料是否齐全
  - 退费计算是否正确
  - 是否有未结案件
  - 凭证状态是否允许退统（ACTIVE状态方可退统）

**步骤6：审核结果**
- 审核通过：进入退费执行环节
- 审核驳回：填写驳回原因，通知分司管理员
- 驳回原因：材料不完整、有未结案件、凭证状态异常、退统原因不成立

**步骤7：生成退费指令**
- 审核通过后，系统生成退费指令
- 退费指令包含：退统车辆信息、退费金额、退费账户
- 退费账户确认：
  - 自有车辆：退至子公司对公账户（分司管理员预设的退费账户）
  - 挂靠车辆（公司代缴）：退至子公司对公账户
  - 挂靠车辆（车主自缴）：退至原缴费账户（车主个人账户）

**步骤8：执行退费**
- 平台财务人员审核退费指令
- 通过企业网银执行转账
- 在系统中记录退费信息：流水号、转账时间、实际退费金额

**步骤9：退费完成**
- 系统更新退统状态
- 通知分司管理员退统已完成

**步骤10：凭证作废**
- 系统将相关统筹单状态更新为CANCELLED（已作废）
- 凭证作废后不再提供任何保障
- 凭证编号保留（不可复用），记录作废原因

**步骤11：记录退统历史**
- 系统记录完整的退统历史：退统原因、退费明细、审核记录、操作人
- 退统记录不可删除（只读归档）
- 分司管理员可在企业端查看历史退统记录

### 6.8 续保流程

**参与方**：系统、分司管理员

**流程说明**：

```
系统                    分司管理员
  │                      │
  ├─ 1.到期前30天提醒 ──>│
  │  (首次提醒)           │
  │                      │
  ├─ 2.到期前15天提醒 ──>│
  │  (二次提醒+费用预估)   │
  │                      │
  ├─ 3.到期前7天提醒 ───>│
  │  (三次提醒+待续保清单) │
  │                      │
  ├─ 4.到期前1天提醒 ───>│
  │  (紧急提醒)           │
  │                      │
  │                      ├─ 5.确认续保清单
  │                      │  (选择续保车辆)
  │<─────────────────────┤
  │                      │
  ├─ 6.生成续保缴费单 ──>│
  │  (按最新费率计算)      │
  │                      │
  │                      ├─ 7.确认并缴费
  │<─────────────────────┤
  │                      │
  ├─ 8.缴费确认          │
  ├─ 9.生成新凭证         │
  ├─ 10.激活新凭证 ─────>│
  │<─ 续保成功通知 ───────┤
```

**详细步骤**：

**步骤1~4：多级到期提醒**

| 提醒时间 | 提醒级别 | 提醒内容 | 发送方式 | 频率 |
|---------|---------|---------|---------|------|
| 到期前30天 | 常规 | 提示有N辆车辆即将到期续保，请及时处理 | 企业端站内信 | 1次 |
| 到期前15天 | 重要 | 列出待续保车辆清单，含最新费率预估 | 企业端站内信 + 短信 | 1次 |
| 到期前7天 | 紧急 | 强调即将到期的车辆数量和未续保风险 | 企业端站内信 + 短信 + 邮件 | 1次 |
| 到期前1天 | 紧急 | 最后提醒，列出即将失去保障的车辆 | 企业端站内信 + 短信 | 1次 |

- 提醒对象：该子公司的所有分司管理员
- 提醒内容包含：待续保车辆数量、车牌号列表、预估费用
- 到期后仍未续保：凭证状态变为EXPIRED（已过期），发送过期通知

**步骤5：确认续保清单**
- 分司管理员在企业端查看待续保车辆列表
- 可全选续保或选择性续保（剔除已报废、转出的车辆）
- 可调整保障方案（如需变更）
- 新增车辆可一并加入续保清单（合并参保）
- 系统按最新费率标准重新计算每辆车的统筹费

**步骤6：生成续保缴费单**
- 系统生成续保缴费单，包含：
  - 续保车辆清单（车牌号、旧凭证号、新费率、新统筹费）
  - 费用汇总（总车辆数、总统筹费、规模折扣、应付金额）
  - 新旧费率对比（若费率有变化）
  - 收款账户信息
  - 缴费截止日期
- 缴费单生成后30天内有效，逾期需重新生成

**步骤7：确认并缴费**
- 分司管理员确认续保缴费单信息
- 通过对公转账完成缴费
- 在企业端上传转账凭证

**步骤8：缴费确认**
- 平台财务人员确认收款
- 系统匹配缴费单和转账记录

**步骤9：生成新凭证**
- 系统为每辆续保车辆生成新的统筹单
- 新凭证生效日：旧凭证到期次日（无缝衔接）
- 新凭证有效期：默认1年（可配置）
- 新凭证编号独立生成（不复用旧编号）

**步骤10：激活新凭证**
- 缴费确认后，系统自动激活新凭证
- 旧凭证状态自动变为EXPIRED
- 新凭证状态为ACTIVE
- 通知分司管理员续保成功

## 七、功能需求详细说明

### 7.1 企业端 Web（核心新增产品）

企业端Web是v2.0的核心产品，面向各子公司的**分司管理员**、**安全员**和**高管**三类角色，提供车辆管理、驾驶员管理、安全管理、案件管理、财务管理等全功能模块。企业端采用响应式Web设计，支持PC和Pad访问。

**三类角色权限概览**：

| 角色 | 核心权限 | 主要使用模块 |
|------|---------|------------|
| **分司管理员** | 本分司日常运营：车辆CRUD、驾驶员绑定/解绑、财务查看/申请、案件查看 | 车辆管理、驾驶员管理、财务管理、案件管理（只看） |
| **安全员** | 本分司安全管理：车辆/驾驶员查看、安全看板、**事故申报审核（必审）** | 安全管理看板、事故申报审核 |
| **高管** | 本企业全局：跨分司数据看板、风险对比、关键审批（退统/大额互助给付确认） | 全局数据面板、退统审批、给付确认 |

#### 7.1.1 企业工作台

**功能描述**：分司管理员登录后的首页，集中展示待办事项、到期提醒、安全评分摘要和出险统计速览，帮助管理员快速掌握当前工作状态。

**功能清单**：

| 功能 | 说明 |
|------|------|
| 待办事项 | 待处理的事故申报确认、给付确认、退统审核、续保确认等 |
| 到期提醒 | 即将到期的统筹单列表（30天内），按紧急程度排序 |
| 安全评分摘要 | 本企业驾驶评分均值、较上月变化、排名 |
| 出险统计速览 | 本月出险次数、未结案件数、本月给付金额 |
| 快捷入口 | 常用功能快捷按钮（车辆管理、事故申报、案件查询等） |
| 最新通知 | 平台发布的最新通知和公告 |

**业务规则**：

1. 待办事项按优先级排序：紧急（红色）> 重要（橙色）> 普通（蓝色），同优先级按时间倒序
2. 到期提醒默认显示未来30天内到期的凭证，支持自定义时间范围
3. 安全评分摘要每日凌晨更新一次
4. 出险统计数据每日凌晨更新一次
5. 待办事项点击后直接跳转到对应功能页面

#### 7.1.2 车辆管理

**功能描述**：分司管理员管理本企业所有自有车辆和挂靠车辆的全生命周期，包括新增、导入、参保、续保、退统等操作。

**功能清单**：

| 功能 | 详细说明 |
|------|---------|
| 车辆列表 | 支持按车牌号/车型/参保状态/车辆归属等条件筛选，支持分页（每页20/50/100条），支持导出Excel |
| 车辆批量导入 | 下载Excel模板 > 填写信息 > 上传 > 系统校验 > 确认导入，单次上限1000条 |
| 新增车辆 | 单辆新增，支持OCR识别行驶证自动填充信息（车牌号、车型、VIN、注册日期等） |
| 车辆详情 | 查看车辆基本信息、参保信息（当前凭证详情）、案件历史、绑定驾驶员、退统记录 |
| 车辆状态监控 | 按参保状态分类统计：在统/即将到期(30天内)/已过期/待续保/欠费停保，支持按状态筛选 |
| 批量续保操作 | 选择多辆即将到期车辆，一键生成续保缴费单 |
| 车辆退统申请 | 选择车辆发起退统申请，填写原因、上传材料，查看退费明细 |
| 自有/挂靠标识 | 列表中明显区分自有车辆和挂靠车辆（标签/图标），挂靠车辆额外显示车主信息 |
| 车辆编辑 | 修改车辆信息（如品牌型号变更、座位数更新等），修改记录留痕 |
| 车辆报废/转出 | 标记车辆状态为已报废或已转出，自动触发相关凭证处理 |

**业务规则**：

1. 同一车牌号在本企业内不可重复添加（唯一性校验）
2. 同一VIN号在本企业内不可重复添加，跨企业通过平台端校验防止重复参保
3. 挂靠车辆参保前必须上传车主签署的《交通安全统筹参与同意书》
4. 车辆状态为ACTIVE（正常运营）才能参保；SCRAPPED（报废）/TRANSFERRED（转出）状态的车辆不可参保
5. 有未结案件的车辆，退统申请需等待案件结案或企业承诺处理完案件后方可提交
6. 批量导入校验不通过的记录，系统提供详细的错误原因说明，分司管理员可修正后重新上传
7. OCR识别行驶证后，分司管理员仍需核对识别结果并手动确认（OCR结果仅供参考）
8. 车辆信息修改后，若影响参保信息（如使用性质变更），系统自动提示可能影响费率

#### 7.1.3 驾驶员管理

**功能描述**：分司管理员管理分配到本企业车辆的驾驶员，包括驾驶员信息查看、驾驶员-车辆绑定/解绑、排班管理等。注意：驾驶员数据挂在集团级驾驶员池中，分司管理员只能进行绑定操作，不能创建或删除驾驶员。

**功能清单**：

| 功能 | 详细说明 |
|------|---------|
| 驾驶员列表 | 显示绑定到本企业车辆的驾驶员列表，支持按姓名/手机号/绑定状态/评分筛选 |
| 驾驶员批量导入 | 下载Excel模板 > 填写驾驶员信息 > 上传 > 系统校验 > 确认导入（导入到集团级驾驶员池） |
| 邀请码加入 | 生成驾驶员邀请码，驾驶员通过小程序输入邀请码加入驾驶员池 |
| 驾驶员详情 | 查看驾驶员基本信息、准驾车型、绑定车辆历史、驾驶评分趋势、出险记录 |
| 驾驶员-车辆绑定 | 从驾驶员池中选择驾驶员绑定到本企业车辆，指定班次（白班/夜班/全天） |
| 驾驶员-车辆解绑 | 解除驾驶员与车辆的绑定关系，记录解绑原因 |
| 白班/夜班排班 | 管理车辆的白班/夜班驾驶员分配，支持排班时间自定义 |
| 驾驶评分查看 | 查看绑定驾驶员的驾驶评分列表和详情 |
| 驾驶员离职处理 | 标记驾驶员为已离职状态，自动解除所有绑定关系 |

**业务规则**：

1. 分司管理员只能绑定集团级驾驶员池中的驾驶员到本企业车辆，不能创建新驾驶员
2. 一辆车最多绑定2名驾驶员（白班+夜班），不能同时绑定多名同班次驾驶员
3. 一名驾驶员同一时间段只能绑定1辆车（轮班场景需先解绑旧车辆再绑定新车辆）
4. 驾驶员离职后，小程序账号冻结（可查看历史数据，不可进行新操作）
5. 驾驶员跨子公司调动时，原子公司先解绑，新子公司再绑定；驾驶评分数据不重置
6. 分司管理员导入驾驶员信息后，系统自动向驾驶员发送短信通知（包含小程序登录方式和初始信息）
7. 绑定/解绑操作需记录操作日志，包含操作人、操作时间、原因
8. 出险时若当班驾驶员未绑定，系统提示分司管理员补充绑定信息

#### 7.1.4 安全管理看板

**功能描述**：为企业提供全面的驾驶安全管理数据视图，帮助分司管理员了解本企业安全状况、识别风险、制定改进措施。

**功能清单**：

| 功能 | 详细说明 |
|------|---------|
| 企业驾驶评分总览 | 本企业驾驶评分均值、较上月变化、评分分布（优秀/良好/一般/较差）、月度趋势折线图 |
| 车队驾驶评分排行 | 按车辆维度展示驾驶评分排行，支持按时间段筛选 |

| 出险统计 | 月度/季度/年度出险次数、出险率、平均给付金额，趋势折线图和同比/环比分析 |
| 出险率对标 | 本企业出险率与行业平均水平、其他子公司出险率的对比分析 |
| 安全培训记录 | 已完成和待完成的安全培训列表，培训参与率统计，驾驶员培训完成情况追踪 |

**业务规则**：

1. 驾驶评分数据每日凌晨更新，展示前一天的数据
2. 风险驾驶员预警列表默认展示评分<70分的驾驶员，分司管理员可自定义阈值
3. 出险率 = 出险次数 / 在统车辆数 x 100%（按月计算）
4. 出险率对标数据由平台端定期更新（月度）
5. 安全培训记录由分司管理员手动录入或平台统一安排后系统自动关联
6. 看板数据默认展示最近12个月的趋势，支持自定义时间范围

#### 7.1.5 案件管理

**功能描述**：分司管理员查看和处理本企业的所有案件，跟踪案件进度，查看出险和赔付统计。

**功能清单**：

| 功能 | 详细说明 |
|------|---------|
| 企业维度案件列表 | 按时间倒序展示本企业所有案件，支持按状态/时间/车辆/金额筛选 |
| 案件详情查看 | 查看案件完整信息：事故申报信息、核查记录、损失评估、材料清单、给付计算、审批记录（时间线形式） |
| 事故申报审核（内部审核） | 企业审核人查看待审核事故申报，进行通过/驳回操作（需企业开启内部审核功能） |

**业务规则**：

1. 分司管理员只能查看本企业关联的案件（通过车辆所属子公司自动关联）
2. 案件列表默认展示最近6个月的案件，支持扩展到全部历史
3. 事故申报审核功能需企业开启"内部审核"开关后才可见，审核人由企业指定
4. 事故申报审核支持多级流转，各级审核人通过后案件才提交至平台
5. 紧急案件（人伤/特急）可配置为自动跳过内部审核
6. 赔付率 = 互助给付支出总额 / 统筹费收入总额 x 100%（按月/季/年计算）
4. 案件详情的时间线按操作时间排序，每条记录显示操作环节、操作人、操作结果
5. 分司管理员可在案件详情中上传补充材料

## 八、驾驶评分体系

### 8.1 数据来源

**重要说明：本版不做GPS实时定位**

v2.0企业版的驾驶评分数据来源于车载终端离线回传数据，而非实时GPS定位。这与v1.2个人版的实时GPS方案有本质区别。

**评分数据来源**：

| 序号 | 数据来源 | 获取方式 | 数据频率 | 数据内容 |
|------|---------|---------|---------|---------|
| 1 | 车载终端离线回传数据 | 车载终端定期（每日/每次熄火后）通过WiFi/4G回传至平台 | 每日或每次行程 | 急刹车次数、急加速次数、急转弯次数、超速次数、超速持续时间、总行驶里程、行驶时长 |
| 2 | 出险历史记录 | 系统内部数据 | 实时（出险后更新） | 近12个月出险次数、赔付金额、责任比例 |
| 3 | 车辆保养记录 | 系统内部数据（手动录入或对接维修系统） | 按次录入 | 保养频率、保养及时性 |
| 4 | 安全培训参与记录 | 系统内部数据（分司管理员录入） | 按次录入 | 培训次数、培训完成率、考试成绩 |

**车载终端数据说明**：

- 车载终端为车辆出厂自带或后装的行车记录仪/车载T-Box设备
- 终端采集驾驶行为事件（急刹、急加速、急转弯、超速等），按行程存储
- 终端在车辆熄火后或到达WiFi热点时自动回传数据至平台
- 回传数据非实时，存在1~24小时延迟
- 平台对接车载终端厂商的API获取数据，需在企业入驻时确认终端型号和对接方式
- 若企业暂无车载终端，驾驶评分仅基于出险记录计算（无安全驾驶行为评分）

### 8.2 评分维度

驾驶评分采用百分制（0~100分），由三个维度加权计算：

```
驾驶评分 = 安全驾驶行为得分 x 60% + 出险记录得分 x 25% + 合规记录得分 x 15%
```

**维度一：安全驾驶行为（60%权重）**

基于车载终端离线回传数据，评估日常驾驶安全水平。

| 事件类型 | 扣分规则 | 说明 |
|---------|---------|------|
| 急刹车 | 每次扣2分 | 减速度超过阈值（默认-6m/s2） |
| 急加速 | 每次扣1.5分 | 加速度超过阈值（默认+4m/s2） |
| 急转弯 | 每次扣2分 | 横向加速度超过阈值（默认+3m/s2） |
| 超速 | 每次扣3分 | 超过限速值的10%以上 |
| 长时间超速 | 每次（持续>30秒）额外扣2分 | 超速持续时间作为加重因子 |

- 基准分：100分
- 每100公里为一个统计周期，计算该周期内的扣分合计
- 月度安全驾驶行为得分 = 100 - 月度平均每百公里扣分（上限0，下限不设）
- 若该月无行驶数据，使用最近3个月的平均值

**维度二：出险记录（25%权重）**

基于近12个月的出险数据，评估事故风险水平。

| 条件 | 扣分 | 说明 |
|------|------|------|
| 每次全责事故 | 扣15分 | 不分金额 |
| 每次主责事故 | 扣10分 | 不分金额 |
| 每次同责事故 | 扣7分 | 不分金额 |
| 每次次责事故 | 扣3分 | 不分金额 |
| 单次给付金额>50,000元 | 额外扣5分 | 不论责任 |

- 基准分：100分
- 仅统计近12个月的出险记录
- 无责事故不扣分
- 出险记录得分 = 100 - 累计扣分（最低0分）

**维度三：合规记录（15%权重）**

基于违章记录和安全培训参与情况。

| 条件 | 扣分 | 说明 |
|------|------|------|
| 每次违章（非超速） | 扣5分 | 如闯红灯、违章停车等 |
| 每次超速违章 | 扣8分 | 交通部门记录的超速违章 |
| 未按时完成安全培训 | 扣3分/次 | 按分司管理员安排的培训计划 |

- 基准分：100分
- 仅统计近12个月的合规记录
- 合规记录得分 = 100 - 累计扣分（最低0分）

**最终评分计算示例**：

| 维度 | 得分 | 权重 | 加权得分 |
|------|------|------|---------|
| 安全驾驶行为 | 85分 | 60% | 51.0分 |
| 出险记录 | 90分 | 25% | 22.5分 |
| 合规记录 | 95分 | 15% | 14.25分 |
| **驾驶评分** | - | - | **87.75分 -> 88分** |

### 8.3 评分展示

**三级看板**：

| 层级 | 展示位置 | 展示内容 | 目标用户 |
|------|---------|---------|---------|
| 个人 | 驾驶员端小程序 | 个人评分（0~100）、评分等级、近6个月趋势、在同车队中的排名（前N%）、各维度得分明细 | 驾驶员 |
| 车队 | 企业端Web | 车队内所有驾驶员的评分排行、评分分布图、高风险驾驶员预警列表（评分<70） | 分司管理员 |
| 企业 | 企业端Web | 企业驾驶评分均值、月度趋势、与行业平均对标、与其他子公司对比 | 分司管理员/高管 |

**评分等级划分**：

| 等级 | 分数范围 | 标识颜色 | 说明 |
|------|---------|---------|------|
| 优秀 | 90~100分 | 绿色 | 安全驾驶水平优秀 |
| 良好 | 80~89分 | 蓝色 | 安全驾驶水平良好 |
| 一般 | 70~79分 | 黄色 | 安全驾驶水平一般，需关注 |

**评分更新频率**：

- 安全驾驶行为得分：每日更新（基于前一天回传的数据）
- 出险记录得分：实时更新（出险结案后24小时内更新）
- 合规记录得分：每周更新（同步违章数据）
- 综合驾驶评分：每日凌晨统一计算

### 8.4 评分与费率联动

企业续约时，企业驾驶评分均值（所有绑定驾驶员的评分算术平均值）作为费率系数的输入因子之一。

**评分与费率联动规则**：

| 企业驾驶评分均值 | 费率调整 | 说明 |
|----------------|---------|------|
| >= 85分 | 费率折扣10% | 安全管理水平优秀，给予激励 |
| 70~84分 | 标准费率（不调整） | 安全管理水平一般 |
| < 70分 | 费率上浮10% | 安全管理水平较差，需风险补偿 |

**应用场景**：

- 企业续约时，系统自动计算该企业过去12个月的驾驶评分均值
- 评分均值作为"企业系数"的计算因子之一
- 费率调整在续约合同中明确体现
- 若企业驾驶评分均值改善显著（如从<70提升至>=85），可协商额外优惠

## 九、费率体系

### 9.1 基础费率

车辆年度基础费根据车型、使用性质、车龄和保障方案综合确定。

**基础费率公式**：

```
车辆年度基础费 = 基准费率 x 车型系数 x 使用性质系数 x 车龄系数 x 保障方案系数
```

**基准费率**：以5座轿车（营运出租）为基准，年度基准费 = 3,000元

**车型系数**：

| 车型类别 | 代表车型 | 车型系数 |
|---------|---------|---------|
| 小型轿车（<=5座） | 捷达、桑塔纳、卡罗拉 | 1.0 |
| 紧凑型轿车（5-7座） | 朗逸、轩逸、速腾 | 1.1 |
| 中型轿车/SUV（7-9座） | 帕萨特、途观、凯美瑞 | 1.3 |
| 大型客车（10-19座） | 考斯特、全顺 | 1.5 |
| 大型客车（20座以上） | 宇通大巴、金龙大巴 | 2.0 |
| 货车（轻型） | 长安之星、五菱荣光 | 1.2 |
| 货车（中型） | 东风天锦、解放J6 | 1.5 |
| 新能源车 | 比亚迪e6、北汽EU5 | 0.9 |

**使用性质系数**：

| 使用性质 | 系数 | 说明 |
|---------|------|------|
| 出租客运 | 1.2 | 日均行驶里程高，风险较高 |
| 网约车/租赁 | 1.1 | 使用频率高 |
| 公路客运 | 1.3 | 长途行驶，风险较高 |
| 旅游客运 | 1.0 | 定线运营，风险适中 |
| 货运 | 1.4 | 货物风险叠加 |

**车龄系数**：

| 车龄 | 系数 | 说明 |
|------|------|------|
| 0~1年 | 0.8 | 新车风险较低 |
| 1~3年 | 0.9 | |
| 3~5年 | 1.0 | 基准 |
| 5~8年 | 1.1 | 老旧车辆风险略高 |
| 8年以上 | 1.3 | 老旧车辆风险较高 |

**保障方案系数**：

| 方案 | 系数 | 说明 |
|------|------|------|
| 基础方案 | 1.0 | 限额较低 |
| 标准方案 | 1.3 | 限额适中 |
| 全面方案 | 1.6 | 限额较高 |

**费率计算示例**：

一辆3年车龄的5座出租车（捷达），选择标准方案：
- 基准费率：3,000元
- 车型系数：1.0（小型轿车）
- 使用性质系数：1.2（出租客运）
- 车龄系数：0.9（1~3年）
- 保障方案系数：1.3（标准方案）
- 车辆年度基础费 = 3,000 x 1.0 x 1.2 x 0.9 x 1.3 = 4,212元

### 9.2 企业系数

企业系数根据企业的风险管理水平综合确定，影响该企业所有车辆的最终费率。

**企业系数公式**：

```
企业系数 = 赔付率因子 x 评分因子 x 续约因子 x 挂靠因子
```

**各因子取值规则**：

| 因子 | 计算方式 | 取值范围 |
|------|---------|---------|
| 赔付率因子 | 基于近12个月赔付率：赔付率<30%取0.85，30%~50%取0.95，50%~70%取1.0，70%~90%取1.15，>90%取1.30 | 0.85 ~ 1.30 |
| 评分因子 | 基于驾驶评分均值：>=85取0.90，70~84取1.0，<70取1.10 | 0.90 ~ 1.10 |
| 续约因子 | 基于续约年限：首次参保取1.0，续约1次取0.98，续约2次取0.95，续约3次及以上取0.92 | 0.92 ~ 1.0 |
| 挂靠因子 | 基于挂靠车辆占比：<=20%取0.98，20%~50%取1.0，50%~80%取1.05，>80%取1.10 | 0.98 ~ 1.10 |

**企业系数示例**：

某企业近12个月赔付率45%（因子0.95），驾驶评分均值78分（因子1.0），首次参保（因子1.0），挂靠车辆占比30%（因子1.0）：
- 企业系数 = 0.95 x 1.0 x 1.0 x 1.0 = 0.95

### 9.3 规模折扣

根据企业在统车辆总数给予阶梯式规模折扣：

| 在统车辆数 | 规模折扣 | 说明 |
|-----------|---------|------|
| < 100辆 | 0%（无折扣） | 小规模企业 |
| 100 ~ 500辆 | 5% | 中小规模企业 |
| 500 ~ 2,000辆 | 10% | 中等规模企业 |
| 2,000 ~ 5,000辆 | 15% | 大规模企业 |
| > 5,000辆 | 20% | 超大规模企业 |

- 规模折扣在参保时根据企业在统车辆总数自动计算
- 年中追加参保不影响已参保车辆的折扣（已缴费部分不退差价）
- 续保时按续保日的在统车辆总数重新计算折扣

### 9.4 最终费率公式

```
车辆年度统筹费 = 车辆年度基础费 x 企业系数 x (1 - 规模折扣)
```

**完整计算示例**：

一辆3年车龄的5座出租车（捷达），选择标准方案，企业系数0.95，在统车辆1,500辆（折扣10%）：

1. 车辆年度基础费 = 3,000 x 1.0 x 1.2 x 0.9 x 1.3 = 4,212元
2. 企业系数 = 0.95
3. 规模折扣 = 10%
4. 车辆年度统筹费 = 4,212 x 0.95 x (1 - 0.10) = 4,212 x 0.95 x 0.90 = 3,601.26元

**取整规则**：最终统筹费四舍五入到整元，即3,601元。

## 十三、风险与控制策略

| 序号 | 风险描述 | 风险等级 | 影响范围 | 控制策略 |
|:----:|---------|:-------:|---------|---------|
| 1 | **车载终端数据对接复杂度高**：不同企业使用不同品牌和型号的车载终端，数据接口和格式不统一，可能导致数据采集困难或数据质量差 | 高 | 驾驶评分准确性 | 首版采用手动录入+文件导入作为兜底方案；优先对接主流终端品牌；建立数据适配层统一数据格式；预留标准API接口供终端厂商对接 |
| 2 | **多租户数据隔离不彻底**：租户间数据泄漏会导致企业间商业信息泄露，造成严重信任危机 | 高 | 全系统 | 采用数据库级别的租户隔离（每条数据强制携带tenant_id，查询时自动过滤）；代码审查重点关注租户隔离；安全测试中增加跨租户访问测试用例；上线前进行渗透测试 |
| 3 | **企业端用户接受度低**：分司管理员习惯原有管理模式，对新系统学习成本高，可能导致使用率低、数据录入不及时 | 中 | 企业端使用效果 | 设计简洁直观的界面交互；提供完整的培训材料和视频教程；安排专人驻场指导首批用户；建立用户反馈快速响应机制；根据反馈持续优化交互体验 |
| 4 | **费率定价偏差**：首版费率模型基于行业经验和假设，实际运营中可能出现定价偏高（客户流失）或偏低（亏损）的情况 | 中 | 平台收入和客户满意度 | 首版采用保守定价策略；预留费率调整机制，运营中按季度评估并调整；费率试算工具支持多种参数组合预览；合同中约定费率调整条款 |
| 5 | **资金池风险**：大额给付或集中出险可能导致某子公司资金池不足，影响给付款支付 | 高 | 给付款支付能力 | 严格执行三账户模型和储备金底线；实时监控资金池余额，预警机制完善；合同中约定大额案件的分摊机制；平台设立应急周转资金 |
| 6 | **驾驶员端小程序使用率低**：驾驶员不主动使用小程序，导致事故申报渠道不通畅、驾驶评分感知度低 | 中 | 事故申报效率和安全管理 | 小程序保持极简设计（仅5个核心页面）；分司管理员在日常管理中引导驾驶员使用；事故申报成功后推送案件进度更新（增加打开率）；驾驶评分提供排名激励 |
| 7 | **合规风险**：统筹业务的法律性质存在灰色地带，可能面临监管质疑或法律诉讼 | 高 | 平台合法运营 | 严格遵守术语规范，全系统不使用保险专属术语；完善非保险告知书和统筹协议；聘请法律顾问定期审查；购买平台责任险；与监管部门保持沟通 |
| 8 | **历史数据迁移**：首批入驻企业可能有历史参保和出险数据需要迁移到新系统，数据格式和质量不可控 | 中 | 首批客户数据完整性 | 提前制定数据迁移方案和模板；安排数据清洗和校验流程；历史数据迁移后进行人工抽检；设置数据过渡期（新旧系统并行） |

## 十五、与v1.2的关系

### 15.1 两版定位对比

| 维度 | v1.2（个人版） | v2.0（企业版） |
|------|---------------|---------------|
| 面向客户 | 出租车驾驶员个人 | 集团化运输企业 |
| 商业模式 | B2C | B2B2C |
| 当前状态 | 已完成PRD，待开发 | 已完成PRD，待开发 |
| 开发优先级 | 后续（预留C端扩张能力） | 优先（当前集团化客户需求） |
| 长期定位 | C端个人市场 | B端企业市场（核心） |

### 15.2 模块共享与独立

| 模块 | v1.2 | v2.0 | 关系 |
|------|------|------|------|
| 数据模型 | 驾驶员-车辆-案件 | 集团-子公司-车辆-驾驶员-案件 | 独立，v2.0为多租户模型 |
| 参保流程 | 个人参保（单辆） | 批量参保（企业级） | 独立，流程差异大 |
| 事故申报流程 | 驾驶员端事故申报 | 多渠道事故申报（驾驶员/企业/平台） | v2.0可复用v1.2事故申报引擎，扩展多渠道 |
| 核查流程 | 现场核查 | 现场核查 | **共享**，流程基本一致 |
| 损失评估 | 损失评估 | 损失评估 | **共享**，流程和审批规则一致 |
| 互助给付 | 个人补偿 | 企业补偿（含企业确认） | v2.0在v1.2基础上增加企业确认环节 |
| 互助理算 | 互助理算公式 | 互助理算公式 | **共享**，核心公式一致，v2.0扩展参数 |
| 驾驶评分 | GPS实时定位 | 车载终端离线数据 | 独立，数据源和算法不同 |
| 费率体系 | 个人评分联动 | 企业系数+规模折扣 | 独立，定价模型完全不同 |
| 财务管理 | 个人收退费 | 三账户+独立结算 | 独立，v2.0复杂度高得多 |
| 现场端App | 核查员App | 核查员App | **共享**，可复用同一套App（租户区分） |
| 管理端Web | 单一后台 | 平台端Web | v2.0平台端功能远超v1.2管理端 |
| 驾驶员端 | 驾驶员App（全功能） | 驾驶员小程序（轻量） | 独立，产品形态和功能范围完全不同 |
| H5 | 验真页 | 验真页+告知书+协议+帮助 | v2.0在v1.2基础上扩展 |
| 通知系统 | 站内信+短信 | 站内信+短信+邮件+小程序推送 | v2.0在v1.2基础上扩展 |

### 15.3 共存策略

**代码层面**：

- 共享模块（核查流程、损失评估、互助理算、现场端App）抽取为独立的服务模块，两版共用
- 独立模块各自开发，通过API网关区分v1.2和v2.0的请求路由
- 数据库层面，v1.2和v2.0使用不同的数据库Schema（或不同数据库实例）

**部署层面**：

- v1.2和v2.0可部署在同一套基础设施上，通过域名和应用标识区分
- 共享服务（如核查服务）独立部署，两版共用
- 独立服务各自部署，互不影响

**运营层面**：

- v1.2面向未来可能的C端个人市场，当前不投入开发资源
- v2.0面向当前集团化客户，为首要开发优先级
- 当v2.0稳定运营后，可根据市场情况决定是否启动v1.2的开发
- 若v1.2启动开发，可复用v2.0的共享模块，减少重复开发

**数据层面**：

- v1.2和v2.0的数据完全隔离，不存在数据互通需求
- 未来若需要数据互通（如集团企业下的驾驶员同时有个人参保需求），需在应用层实现跨版数据关联
